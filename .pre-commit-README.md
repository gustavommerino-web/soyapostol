# Pre-commit quality gate

This repo uses [pre-commit](https://pre-commit.com/) to catch code issues
*before* they reach a commit or a reviewer. Agreed bar:

| Tool           | Scope                | Rule                  |
|----------------|----------------------|-----------------------|
| **ruff**       | `backend/**.py`      | `--select F,E9,W6`    |
| **eslint**     | `frontend/src/**/*`  | `--max-warnings=0`    |
| **examen-privacy** | `Examen.jsx`     | no network calls      |
| hygiene hooks  | whole repo           | trailing ws / EOF / merge markers / big files |

## One-time install (per clone)

```bash
pip install pre-commit
pre-commit install
```

`pre-commit install` writes a git hook at `.git/hooks/pre-commit`. From then
on every `git commit` runs the above tools on staged files and aborts the
commit if any finding surfaces.

## Run manually (CI / audit / after a rebase)

```bash
pre-commit run --all-files
```

First run will fetch the pinned ruff + hygiene repos (≈30s); subsequent runs
are local and fast.

## Why F/E9/W6 specifically?

* `F`  — pyflakes: undefined names, unused imports, `is`/`==` misuse,
         string-format errors → the **real** bugs.
* `E9` — syntax / indentation / bare-except → won't even run otherwise.
* `W6` — deprecation warnings (e.g. `datetime.utcnow()` vs
         `datetime.now(timezone.utc)`).

Everything else (style, line length, docstring formatting) is out of scope
on purpose. The bar catches real defects; it doesn't nag.

## Skipping a hook (rare, document in PR)

```bash
SKIP=eslint-frontend git commit -m "hotfix: ..."
```

If you skip, the next `pre-commit run --all-files` in CI will still catch
whatever you skipped — so nothing actually escapes.
